# 数据工程：从原始网页到训练语料

> 模型架构趋同，数据才是护城河。GPT-4 之后，每家都在卷数据而不是卷架构。

---

## 一、数据工程全流程

```
原始网页 (Common Crawl, 百亿级)
    ↓
[1. 语言识别 / 过滤]      扔掉非目标语言、乱码
    ↓
[2. 质量过滤]             扔掉低质量内容（太短、重复、广告）
    ↓
[3. 去重]                 文档级 + 段落级 + 句子级
    ↓
[4. PII 脱敏]             去姓名、电话、邮箱
    ↓
[5. Data Mixture]         各领域配比
    ↓
[6. Tokenization]         文本 → token 序列
    ↓
训练语料
```

| 来源 | 规模 | 质量 | 代表性模型 |
|------|------|------|-----------|

## 三、质量过滤 — 怎么把垃圾扔掉

### 3.1 规则过滤（快，粗筛）

```python
# 一条文本被判为垃圾如果：
- 长度 < 100 字符                    # 太短没信息
- 非字母比例 > 50%                   # 乱码
- 平均词长 > 15 字符                 # 异常
- 停用词比例 < 10%                   # 不正常（可能是 HTML dump）
- 困惑度 > 阈值（用 fastText 等）     # 不符合自然语言
- 包含大量重复 n-gram               # SEO 垃圾
```

### 3.2 模型过滤（慢，精筛）

用轻量分类器给文本打分：
- **fastText**：训练一个二分类器（Wikipedia vs Common Crawl），Wikipedia 为正例
- **KenLM 困惑度**：5-gram 语言模型，困惑度太高 = 不是正常文本
- **BERT 质量打分**：FineWeb 用 BERT 给每条文本打分

### 3.3 FineWeb 的做法（HuggingFace 开源）

```
1. 规则粗筛 → 扔掉明显垃圾
2. fastText 二分类 → 保留像 Wikipedia 的
3. KenLM 困惑度过滤 → 保留语言通顺的
4. 去重 → MinHash + URL 去重
```

## 四、去重 — 怎么删掉重复内容

### 4.1 为什么要去重？

- 训练数据重复 → 过拟合 → 模型背答案而不是理解
- 验证集污染：训练集和测试集有重叠 → 评估失真
- 浪费算力：重复数据吃 token budget 却没带来新知识

### 4.2 三级去重策略

| 级别 | 方法 | 效果 |
|------|------|------|
| URL 去重 | 哈希 URL | 去掉完全相同的网页 |
| 文档级去重 | MinHash LSH | 去掉内容相似的文档（如转载） |
| 段落/句子级去重 | Exact match + fuzzy | 去掉模板话术、网站 footer 等 |

### 4.3 MinHash 原理（核心面试点）

> MinHash 是一种快速估算两个集合 Jaccard 相似度的方法。

```python
# 直觉：
# 文档 A = {the, cat, sat, on, mat}
# 文档 B = {the, cat, slept, on, mat}
# Jaccard = |A∩B| / |A∪B| = 4/6 = 0.67

# 百万文档两两比较太慢 → MinHash:
# 对每个文档生成短签名（如 128 个哈希值）
# 两文档的 MinHash 签名相似 ≈ Jaccard 相似 → 聚类去重
```

### 4.4 各模型去重实践

| 模型 | 做法 |
|------|------|
| GPT-3 | MinHash LSH（文档级）+ 模糊去重 |
| LLaMA | URL + MinHash + 行级去重 |
| DeepSeek | 激进去重：MinHash + 语义去重（用嵌入距离） |
| DCLM | 极致去重：文档 + 段落 + 句子三级全上 |

## 五、Data Mixture — 怎么配比

### 5.1 核心思想

不是所有数据价值相等。Wikipedia 的知识密度远高于随机网页。各领域按一定比例混合，让模型全面学习。

### 5.2 LLaMA 的配比

| 数据源 | 占比 |
|--------|------|
| Common Crawl | 67.0% |
| C4 | 15.0% |
| GitHub | 4.5% |
| Wikipedia | 4.5% |
| Books | 4.5% |
| ArXiv | 2.5% |
| Stack Exchange | 2.0% |

> 代码只有 4.5%，但大大提升了推理能力。说明少量高质量代码数据对 reasoning 很重要。

### 5.3 动态配比 vs 静态配比

- **静态**：固定比例，简单但可能浪费（某类数据提前学完）
- **动态**：训练过程中调整。比如 DeepSeek 的做法：根据各类数据的 loss 下降速度，自动增减该类的采样权重

## 六、Token Budget — 数据要喂多少

### 6.1 Chinchilla 最优配比

$$D \approx 20 \times N$$

| 参数量 | 建议 token 数 |
|--------|-------------|
| 1B | 20B tokens |
| 7B | 140B tokens (≈ 1000 亿中文) |
| 13B | 260B tokens |
| 70B | 1.4T tokens |

### 6.2 实际模型都"超喂"

| 模型 | 参数量 | 训练 token 数 | D/N 比 |
|------|--------|-------------|--------|
| Chinchilla (理论) | 70B | 1.4T | 20x |
| LLaMA 1 | 65B | 1.4T | 22x |
| LLaMA 2 | 70B | 2T | 29x |
| LLaMA 3 | 70B | 15T | **214x** |
| DeepSeek-V3 | 671B (37B 激活) | 14.8T | — |

> **趋势：数据多多益善。** Chinchilla 是最低配比，实际训练都远高于此。高质量数据不会过拟合。

## 七、面试必背

### Q1：数据工程为什么重要？

> 模型架构趋同（都是 Decoder-only Transformer），数据质量决定模型上限。GPT-4 之后，竞争焦点从架构创新转向数据工程。好的数据 = 更好的效果 + 更少的算力。

### Q2：怎么去重？MinHash 是什么？

> MinHash 是一种快速估算集合 Jaccard 相似度的算法。对文档的 n-gram 集合做多次哈希，取最小值作为签名。两文档签名的重合度近似于 Jaccard 相似度，结合 LSH 可以快速聚类去重。实际训练中通常做三级去重：URL → 文档级 MinHash → 句级去重。

### Q3：怎么决定数据配比？

> 先参考已有模型的配比（LLaMA 的 67% CC + 其他），再根据下游任务调整——代码多给点能提升推理，书籍多给点能提升知识。更进一步可以动态配比：根据各类数据的 loss 下降速度自动调整采样权重。

### Q4：要喂多少数据？

> Chinchilla 的最优配比是 D ≈ 20N，但这只是最低建议。实际上 LLaMA 3 喂到了 214 倍参数量的数据，效果仍然在涨。高质量数据多多益善。

## 八、总结

```
数据处理管道:
  原始数据
    → 语言过滤（扔掉非目标语言）
    → 质量过滤（规则 + 模型打分）
    → 去重（URL → MinHash → 句级）
    → PII 脱敏
    → 配比混合
    → Tokenize
    → 训练
```

> 三个核心原则：**垃圾进垃圾出、避免过拟合（去重）、各领域平衡（data mixture）。**
